# Phase 1.4 — Google OAuth Login + WeChat Channel Binding

> **For agentic workers:** Execute task-by-task. Each task is small (2-5 min). After each task: run the named tests, commit only if green, then proceed.

**Goal:** Replace dev-only login with Google OAuth via a **provider registry** (so adding GitHub/Apple/微信开放平台 later is one file + one row, no route changes). Then replace `WechatApp` placeholder with real 微信公众号测试号 scan-to-bind channel, gated by `container_mapping.status='ready'`.

**Architecture:**
- **A. Login** — `OAuthProvider` interface + provider registry. Single dynamic Express route `/api/auth/:provider/start` and `/api/auth/:provider/callback` dispatches to the right provider. Identities stored in new `oauth_identities(provider, external_id) → user_id` table, so one user can bind multiple identities later. Path-based callbacks chosen over query-param for max OAuth-platform compatibility (Apple/Microsoft don't allow query).
- **B. Binding** — 微信公众号测试号 `qrcode/create` API issues a scene_str-tagged QR; user scans → follows → 微信 sends webhook `subscribe`/`SCAN` event; backend matches scene_str → openid mapping written to `wechat_bindings`. Frontend polls a status endpoint. Precondition for `/bind/start`: caller's container must be `ready`. ngrok exposes the webhook publicly during dev.
- **Independence guarantee** — A and B share NOTHING: different external services (Google vs 微信), different config namespaces (`auth.providers` vs `wechatMp`), different tables, different routes, different UI entry points, different test files. Either side can be broken/disabled without affecting the other.

**Tech Stack additions:**
- Google OAuth 2.0 (`accounts.google.com/o/oauth2/v2/auth` + `oauth2.googleapis.com/token` + `openidconnect.googleapis.com/v1/userinfo`)
- 微信公众平台测试号 API (`api.weixin.qq.com/cgi-bin/{token,qrcode/create}` + webhook XML parsing + sha1 signature)
- ngrok (dev only, webhook public URL)
- New env vars: `GOOGLE_CLIENT_ID/SECRET/REDIRECT_URI`, `WECHAT_MP_APPID/SECRET/TOKEN`, `PUBLIC_BASE_URL` (for building absolute redirect URIs and ngrok-aware webhook)

CREATE TABLE wechat_bindings (

### B5. wechat-bind API routes

**Files:**
- Create: `apps/gateway/src/api/wechat-bind.ts`
- Test: `apps/gateway/test/api/wechat-bind.test.ts`

4 endpoints:

- `POST /api/wechat/bind/start` (sessionMw):
  - precondition: `container_mapping.status === 'ready'` for this user, else 412
  - precondition: no existing wechat_binding for this user, else 409
  - generate `token = randomBytes(16).hex`, call `createBindQrCode(scene_str = 'bind_' + token)`, insert into wechat_bind_tickets, return `{qr_url, token, expires_at}`

- `GET /api/wechat/bind/status?token=xxx` (sessionMw):
  - query wechat_bind_tickets by token + user_id; if `mp_openid IS NOT NULL` AND wechat_bindings now contains the row → `{bound:true, mp_openid}`; else `{bound:false}`

- `GET|POST /api/wechat/webhook` (NO sessionMw):
  - `GET`: 微信 sends `{signature, timestamp, nonce, echostr}`; verify signature → reply `echostr` body
  - `POST`: 微信 sends XML event;
    - verify signature
    - parse event
    - if `subscribe` or `SCAN`: parse scene_str (strip `qrscene_` prefix if present), look up wechat_bind_tickets by token; if found and not expired and user is the right one → INSERT into wechat_bindings (ON CONFLICT DO NOTHING; idempotent for retries) → UPDATE wechat_bind_tickets SET mp_openid=... WHERE token=...
    - always respond `<xml>...success</xml>` or empty 200 (无论事件类型,微信会重试 non-200)

- `DELETE /api/wechat/bind` (sessionMw):
  - delete from wechat_bindings where user_id; also delete pending tickets. Return `{ok:true}`. Does NOT 取关 (we can't, that's the user's WeChat action).

Tests: precondition failure, status polling, webhook signature, subscribe event flow end-to-end, SCAN event, replay idempotency (same event twice → single row), unbind.
